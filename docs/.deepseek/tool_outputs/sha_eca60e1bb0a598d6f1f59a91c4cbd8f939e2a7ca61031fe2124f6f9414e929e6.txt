   104│ 首先，需要解释一下用户命名空间（User Namespace）的工作原理：它允许将容器内的用户ID（UID）和组ID（GID）映射到宿主机上的不同UID/GID。通常，容器运行时（如Docker）会创建一个用户命名空间，并设置映射关系。常见的映射是：容器内的root（UID 0）被映射到宿主机上的一个非特权用户（比如UID 100000，或者dockremap用户），容器内的其他用户ID会按照偏移量进行映射。
   105│ 
   106│ 但是，需要注意：并不是所有容器都默认启用用户命名空间。在Docker中，用户命名空间默认是禁用的，需要通过配置开启。很多容器直接使用宿主机的UID命名空间，即容器内的root就是宿主机的root（这有安全风险）。所以实际情况取决于配置。
   107│ 
   108│ 不过，问题假设了“容器内的root用户是宿主机的普通用户”，这暗示已经启用了用户命名空间，并且设置了映射关系，比如容器内UID 0映射到宿主机UID 1000（普通用户）。那么问题就是：容器内的普通用户（比如UID 1000）在宿主机上对应什么？
   109│ 
   110│ 在用户命名空间映射中，通常有一个映射范围。比如，可以设置将容器内的UID 0-65535映射到宿主机上的UID 100000-165535。这样容器内UID 0对应宿主机UID 100000，容器内UID 1对应宿主机UID 100001，以此类推。如果容器内有一个普通用户UID 1000，那么它在宿主机上就对应UID 100000+1000 = 101000。这个UID在宿主机上可能并不存在对应的用户账号（即/etc/passwd中没有），但仍然是一个有效的UID，内核只认数字。进程以该UID运行，对文件的访问权限也取决于文件的属主UID。
   111│ 
   112│ 因此，答案应该是：容器内的普通用户会被映射到宿主机上的一个高位的UID（根据映射规则），这个UID在宿主机上通常没有对应的用户名，只是一个数字ID。它的权限范围仅限于该UID拥有的资源，并且受到宿主机文件系统权限的控制。
   113│ 
   114│ 如果映射规则是线性的，那么容器内的UID x对应宿主机上的UID base + x。但有些映射可能不是线性的，或者只有root被映射。通常Docker的用户命名映射是：将容器内的UID 0映射到宿主机上的一个非root用户（比如在/etc/subuid中定义的），然后将容器内的其他UID映射到从某个起始值开始的范围。默认情况下，Docker会为每个用户分配一个ID范围（/etc/subuid中定义），比如dockremap:100000:65536，表示将容器内0-65535映射到宿主机100000-165535。
   115│ 
   116│ 所以，对于容器内的普通用户，它对应宿主机上的一个超出常规范围的UID，通常没有登录shell和家目录，但可以拥有文件。
   117│ 
   118│ 回答时，可以结合这个原理，并提醒用户命名空间不一定总是启用。
   119│ 
   120│ 现在组织答案。
   121│ 在启用了用户命名空间（User Namespace）的 Linux 容器中，**容器内的 root 用户被映射为宿主机上的一个普通用户（非特权 UID）**，这是通过 UID 映射机制实现的。那么，**容器内的普通用户（例如 UID=1000）在宿主机上对应什么？**
   122│ 
   123│ ---
   124│ 
   125│ ### 答案：宿主机上的一个高位 UID（数字 ID），通常无对应用户名
   126│ 
   127│ 具体来说：
   128│ 
   129│ 1. **映射规则是线性的**  
   130│    容器运行时（如 Docker 配置了 `userns-remap`）会从 `/etc/subuid` 文件中读取一段预留的 UID 范围，例如：  
   131│    ```
   132│    dockremap:100000:65536
   133│    ```  
   134│    这表示将容器内的 UID 0~65535 **依次映射**到宿主机的 UID 100000~165535。  
   135│    - 容器内 UID 0 → 宿主机 UID 100000  
   136│    - 容器内 UID 1 → 宿主机 UID 100001  
   137│    - 容器内 UID 1000 → 宿主机 UID **100000 + 1000 = 101000**
   138│ 
   139│ 2. **宿主机上只有数字 ID，没有用户名**  
   140│    这个 UID（如 101000）在宿主的 `/etc/passwd` 中通常没有对应的条目，因此 `ls -l` 显示文件属主时只会显示数字。但内核完全基于 UID 进行权限检查，进程以该 UID 运行，对文件的访问权限也取决于该 UID 是否拥有相应权限。
   141│ 
   142│ 3. **实际权限受限于映射范围**  
   143│    这个高位 UID 在宿主机上是一个普通用户级别的 ID，没有特权。它只能访问自己拥有的文件（那些属主 UID 为 101000 的文件），以及被其他用户或组授予权限的资源。
   144│ 
   145│ ---
   146│ 
   147│ ### 特殊情况说明
   148│ 
   149│ - 如果容器内没有启用用户命名空间（默认情况），容器内的 UID 直接就是宿主机的 UID，那么容器内的 root 就是宿主机的 root，容器内的普通用户就是宿主机的同名用户（如果存在）。
   150│ - 如果容器内启用了用户命名空间，但只映射了 root（比如某些特殊配置），那么容器内非 root 用户可能无法正常工作，因为其 UID 在宿主机上没有对应映射，会被当作宿主机上的“nobody”或超出范围的 ID 处理。
   151│ 
   152│ ---
   153│ 
   154│ ### 总结
   155│ 
   156│ 容器内的普通用户通过 UID 映射机制，对应到宿主机上一个**高位的、未分配的数字 ID**。这个设计既保持了容器内用户权限的直观性（如普通用户无法执行特权操作），又通过隔离的 ID 空间增强了安全性。
