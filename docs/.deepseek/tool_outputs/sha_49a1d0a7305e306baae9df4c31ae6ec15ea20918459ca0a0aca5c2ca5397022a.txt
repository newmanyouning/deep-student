   323│ # 全国标准信息爬虫增量更新与缓存系统开发需求
   324│ 
   325│ ## 一、项目目的
   326│ 
   327│ 在当前爬虫项目（已获得网站API，脚本以字段形式返回爬取信息）基础上，新增“增量更新检测”与“本地缓存机制”，避免对未变化的标准条目重复执行完整爬取与Excel写入操作。
   328│ 
   329│ ## 二、背景调研摘要
   330│ 
   331│ 根据对目标网站及其数据服务接口的调研：

   333│ 1. **目标网站更新机制**：全国标准信息公共服务平台（https://std.samr.gov.cn/）的标准化大数据服务接口（https://www.sacinfo.cn/ncseAuDb）明确支持“全量数据获取”和“增量数据获取”两种模式，可第一时间获取标准数据[reference:0]。国家标准批准发布后20个工作日内公开标准文本[reference:1]。
   334│ 
   335│ 2. **增量爬取通用方案**：业界主流增量检测方式包括：基于时间戳比对、基于内容哈希（MD5/SHA1）比对、基于HTTP条件请求（ETag / Last-Modified / If-Modified-Since）比对[reference:2][reference:3]。

   339│ ## 三、方案设计
   340│ 
   341│ ### 3.1 缓存存储设计
   342│ 
   343│ 在项目根目录下创建本地缓存文件，采用 **SQLite 数据库**（文件名为 `std_cache.db`）作为缓存存储，设计一张缓存表：
   344│ 
   345│ ```sql
   346│ CREATE TABLE IF NOT EXISTS std_cache (
   347│     id              TEXT PRIMARY KEY,      -- 标准唯一标识（API返回的字段中能唯一确定一条标准的ID）
   348│     content_hash    TEXT NOT NULL,         -- 爬取信息关键字段拼接后的哈希值（如SHA256）
   349│     last_modified   TEXT,                  -- API返回的更新时间字段（如有）
   350│     etag            TEXT,                  -- HTTP响应ETag（如有）
   351│     cached_at       TEXT NOT NULL,         -- 本地缓存时间（ISO 8601格式）
   352│     raw_response    TEXT                   -- 可选：缓存原始API响应JSON，便于调试
   353│ );
   354│ ```
   355│ 
   356│ **选型理由**：SQLite无需额外服务进程，读写轻量；单文件部署，与现有Python项目天然兼容；支持复杂查询，便于后续扩展。
   357│ 
   358│ ### 3.2 增量更新流程
   359│ 
   360│ 整个爬虫执行流程设计如下：
   361│ 
   362│ ```
   363│ 1. 程序启动，加载本地缓存（std_cache.db）
   364│ 2. 调用网站API，获取标准列表/搜索分页结果
   365│ 3. 对每条返回的标准数据：
   366│    a. 提取唯一标识（id）
   367│    b. 根据API当前返回的关键字段，计算 content_hash
   368│    c. 查询本地缓存中是否存在该id
   369│    d. 若不存在 → 标记为【新增】，执行爬取并写入Excel
   370│    e. 若存在且 content_hash 不同 → 标记为【更新】，执行爬取并更新Excel中对应行
   371│    f. 若存在且 content_hash 相同 → 跳过，无需重复写入
   372│ 4. 所有分页/条目处理完毕后，更新本地缓存（插入新记录、更新已有记录）
   373│ 5. 保存Excel文件
   374│ ```
