   208│ ## 二、核心优化策略（按效果从大到小排序）
   209│ 
   210│ ### 1. 使用**增量上下文**而非全量项目快照
   211│ 大多数Agent框架默认每次都将整个项目目录作为上下文发送。这是最大的浪费。
   212│ 
   213│ **解决方案**：
   214│ - **基于Git diff的上下文**：每次只发送自上次成功构建以来的**文件变更**。Agent只需要看到修改的部分，不需要重新读取未变文件。
   215│ - **实现方式**：在Agent循环中，调用`git diff HEAD~1`获取变更，将这些diff发给AI，要求生成基于当前代码的补丁（patch），而不是全量文件。
   216│ 
   217│ **效果**：每次迭代输入 token 从 50k 降到 **2k~5k**（节省90%）。
   218│ 
   219│ ### 2. 分离**规划与执行**，减少无效重写
   220│ 全自动Agent常见的坏习惯：遇到一个错误就重写整个模块。
   221│ 
   222│ **解决方案**：
   223│ - **先规划后执行**：Agent先输出一份“修复计划”（100-300 token），你（或本地验证器）判断计划是否合理。合理后才让Agent输出具体代码。
   224│ - **使用`plan-and-solve`提示**：强制Agent输出`[PLAN]`和`[CODE]`两个部分。本地脚本可以只提取`[CODE]`，并且如果两次计划相同，则终止循环。
   225│ - **设置最大重试次数**：针对同一个错误，Agent最多尝试3次不同的修复，如果都不行，则上报人工。
   226│ 
   227│ **效果**：减少80%的无效代码生成。
   228│ 
   229│ ### 3. 本地预处理测试反馈，只发送**最小必要信息**
   230│ 测试输出可能非常长（例如pandas的警告、DataFrame的repr输出）。全发给AI是巨大的浪费。
   231│ 
   232│ **解决方案**：
   233│ - 编写一个本地**日志过滤器**：
   234│   - 只提取**失败断言的位置**（文件名、行号、错误类型）
   235│   - 忽略所有`UserWarning`、`DeprecationWarning`（除非导致失败）
   236│   - 如果测试输出超过2000字符，只发送前500字符和后500字符
   237│ - 对于Excel分析项目，**不要让AI看到实际的Excel数据**，只发送数据的概要（列名、数据类型、行数、缺失值比例）。
   238│ 
   239│ **效果**：测试反馈 token 从 50k 降到 **1k~3k**（节省95%）。
   240│ 
   241│ ### 4. 文件拆分重构：使用**本地依赖分析 + 只让AI生成拆分方案**
   242│ Agent在重构时，往往需要读取所有文件才能决定拆分边界，这导致一次重构消耗几十万token。
   243│ 
   244│ **解决方案**：
   245│ - **本地自动检测文件大小**：当任何`.py`文件超过300行时，由本地脚本触发重构请求，而不是由Agent自己发现。
   246│ - **只让AI生成拆分映射**：将大文件的代码发给AI（只发该文件，不发整个项目），要求输出“函数A应该移动到module_x.py，函数B保留”这样的映射。本地脚本根据映射执行实际的移动和修改import。
