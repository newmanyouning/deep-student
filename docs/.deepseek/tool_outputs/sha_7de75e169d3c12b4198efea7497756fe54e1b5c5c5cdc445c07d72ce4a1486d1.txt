   208│ ### 1.3 注意力机制：模型如何“聚焦”
   209│ 
   210│ 注意力机制是Transformer架构的核心创新，允许模型在处理每个Token时，动态地评估上下文中所有其他Token的重要性，并据此加权组合信息。
   211│ 
   212│ **自注意力计算流程：**
   213│ 
   214│ 1. **Query、Key、Value向量生成**：每个输入Token通过三个不同的权重矩阵转换为Query(Q)、Key(K)、Value(V)向量。
   215│ 
   216│ 2. **注意力分数计算**：对于每个Token，计算其Query与所有Token的Key的点积，得到注意力分数。
   217│    ```
   218│    Attention(Q,K) = softmax(Q·K^T / √d_k)
   219│    ```
   220│ 
   221│ 3. **加权求和**：将注意力分数作为权重，对所有Token的Value向量进行加权求和，得到该位置的新表示。
   222│ 
   223│ **多头注意力机制：**
   224│ 
   225│ ```python
   226│ # 概念性伪代码
   227│ class MultiHeadAttention:
   228│     def __init__(self, num_heads=8, d_model=512):
   229│         self.num_heads = num_heads
   230│         self.d_model = d_model
   231│         self.head_dim = d_model // num_heads
   232│         
   233│     def forward(self, query, key, value):
   234│         # 1. 线性投影并分割成多头
   235│         Q = self.split_heads(self.W_q(query))  # [batch, seq_len, num_heads, head_dim]
   236│         K = self.split_heads(self.W_k(key))
   237│         V = self.split_heads(self.W_v(value))
   238│         
   239│         # 2. 并行计算每个头的注意力
   240│         attention_outputs = []
   241│         for i in range(self.num_heads):
   242│             scores = torch.matmul(Q[:, :, i, :], K[:, :, i, :].transpose(-2, -1))
   243│             scores = scores / math.sqrt(self.head_dim)
   244│             weights = F.softmax(scores, dim=-1)
   245│             output = torch.matmul(weights, V[:, :, i, :])
   246│             attention_outputs.append(output)
   247│         
   248│         # 3. 合并多头输出
   249│         concatenated = torch.cat(attention_outputs, dim=-1)
   250│         return self.W_o(concatenated)
   251│ ```
   252│ 
   253│ **注意力模式对提示词设计的影响：**
   254│ 
   255│ | 注意力模式 | 特征 | 提示词策略 |
   256│ |-----------|------|-----------|
   257│ | 局部聚焦 | 关注邻近Token | 重要信息放在同一句子/段落内 |
   258│ | 全局聚焦 | 关注远距离依赖 | 可使用长距离引用，如"如前所述" |
   259│ | 重复抑制 | 对重复Token注意力降低 | 避免不必要的信息重复 |
   260│ | 位置偏见 | 对开头/结尾注意力更强 | 关键指令放在开头或结尾 |
   261│ 
   262│ > **专家提示**：研究表明，LLM对提示词开头和结尾的注意力权重平均高出中间部分30-50%。将最重要的约束、示例或问题放在提示词的开头和结尾，中间部分放置补充说明。

   264│ ### 1.4 上下文窗口：记忆的边界
   265│ 
   266│ 上下文窗口是模型一次能处理的最大Token数量（输入+输出）。例如，GPT-4 Turbo支持128K Token，Claude 3支持200K Token。
   267│ 
   268│ **上下文窗口的层级利用：**
   269│ 
   270│ ```
   271│ Level 1 (0-4K): 核心指令和示例
   272│ ├── 系统提示词
   273│ ├── 当前任务指令
   274│ └── 2-3个关键示例
   275│ 
   276│ Level 2 (4K-32K): 补充材料
   277│ ├── 相关背景文档
   278│ ├── 知识库片段
   279│ └── 历史对话摘要
   280│ 
   281│ Level 3 (32K-128K): 扩展参考
   282│ ├── 完整文档
   283│ ├── 代码库
   284│ └── 长对话记录
   285│ ```
   286│ 
   287│ **上下文窗口饱和的症状：**
   288│ - 模型开始"遗忘"提示词早期部分的内容
   289│ - 输出变得重复或发散
   290│ - 无法遵循复杂指令
   291│ - API返回最大长度错误
   292│ 
   293│ **上下文窗口管理策略：**
   294│ 
   295│ ```markdown
   296│ ## 错误示例：信息过载
   297│ 用户: 这是一本300页小说的完整文本... (150K Token) 
   298│       请分析主角的性格特点。
   299│ 
   300│ 模型: (错误或截断输出)
   301│ 
   302│ ## 改进示例：分块+摘要
   303│ 用户: 请基于以下小说摘要分析主角性格：
   304│       
   305│       ## 第1-3章摘要
   306│       主角张三出身贫寒，性格倔强...
   307│       
   308│       ## 第4-6章摘要  
   309│       面对挫折，张三表现出坚韧和创造力...
   310│       
   311│       ## 第7-10章摘要
   312│       成功后的张三开始显露傲慢...
   313│       
   314│       问题: 张三的性格是如何演变的？请列举3个关键转折点。

   317│ ### 1.5 关键参数详解
   318│ 
   319│ #### 1.5.1 Temperature（温度）
   320│ 
   321│ 控制输出随机性的参数，取值范围通常为0-2。
   322│ 
   323│ | Temperature值 | 行为特征 | 适用场景 |
   324│ |--------------|---------|----------|
   325│ | 0 - 0.3 | 确定性高，重复性强 | 代码生成、数学解题、数据提取 |
   326│ | 0.4 - 0.7 | 平衡创造性和准确性 | 一般对话、内容改写、翻译 |
   327│ | 0.8 - 1.2 | 高创造性，多样性好 | 创意写作、头脑风暴 |
   328│ | 1.3 - 2.0 | 高随机性，可能产生幻觉 | 实验性生成、艺术创作 |
   329│ 
   330│ **Temperature的工作机制：**
   331│ 
   332│ ```python
   333│ import numpy as np
   334│ 
   335│ def apply_temperature(logits, temperature):
   336│     """
   337│     logits: 模型输出的原始分数
   338│     temperature: 温度参数
   339│     """
   340│     if temperature == 0:
   341│         # 退化为argmax（完全确定性）
   342│         return np.argmax(logits)
   343│     
   344│     # 缩放logits并应用softmax
   345│     scaled_logits = logits / temperature
   346│     probabilities = np.exp(scaled_logits) / np.sum(np.exp(scaled_logits))
   347│     
   348│     # 根据概率分布采样
   349│     return np.random.choice(len(logits), p=probabilities)
   350│ 
   351│ # 示例：初始logits假设为[2.0, 1.0, 0.5]
   352│ logits = np.array([2.0, 1.0, 0.5])
   353│ 
   354│ # Temperature=0.2 (低随机性)
   355│ probs_t02 = apply_temperature(logits, 0.2)
   356│ # 输出: 几乎总是第0个token (概率>99%)
   357│ 
   358│ # Temperature=1.0 (标准)
   359│ probs_t10 = apply_temperature(logits, 1.0)
   360│ # 输出: 约65%概率第0个token, 26%第1个, 9%第2个
   361│ 
   362│ # Temperature=1.5 (高随机性)
   363│ probs_t15 = apply_temperature(logits, 1.5)
   364│ # 输出: 约50%概率第0个token, 30%第1个, 20%第2个
   365│ ```
   366│ 
   367│ #### 1.5.2 Top-p (Nucleus Sampling)
   368│ 
   369│ 动态选择概率质量总和达到p的最小token集合进行采样。
   370│ 
   371│ ```python
   372│ def top_p_sampling(probabilities, p=0.9):
   373│     """
   374│     probabilities: 按降序排列的概率数组
   375│     p: 累积概率阈值
   376│     """
   377│     sorted_indices = np.argsort(probabilities)[::-1]
   378│     sorted_probs = probabilities[sorted_indices]
   379│     
   380│     # 计算累积概率
   381│     cumsum_probs = np.cumsum(sorted_probs)
   382│     
   383│     # 找到达到阈值p的位置
   384│     cutoff_index = np.where(cumsum_probs >= p)[0][0] + 1
   385│     
   386│     # 保留前cutoff_index个token
   387│     selected_indices = sorted_indices[:cutoff_index]
   388│     
   389│     # 重新归一化并采样
   390│     filtered_probs = probabilities[selected_indices]
   391│     filtered_probs = filtered_probs / np.sum(filtered_probs)
   392│     
   393│     return np.random.choice(selected_indices, p=filtered_probs)
   394│ ```
   395│ 
   396│ **Temperature vs Top-p 对比：**
   397│ 
   398│ | 场景 | 推荐配置 | 原因 |
   399│ |------|---------|------|
   400│ | 精确任务 | T=0, Top-p=1.0 | 完全确定性输出 |
   401│ | 创意写作 | T=0.8, Top-p=0.95 | 平衡创造性和合理性 |
   402│ | 代码生成 | T=0.2, Top-p=0.9 | 减少语法错误 |
   403│ | 多答案生成 | T=0.7, Top-p=0.8 | 在多样性和质量间权衡 |
   404│ 
   405│ #### 1.5.3 其他重要参数
   406│ 
   407│ **Frequency Penalty (频率惩罚)：**
   408│ - 取值范围：-2.0 到 2.0
   409│ - 正值：减少重复使用已出现的Token
   410│ - 负值：鼓励使用已出现的Token
   411│ - 公式：`score -= frequency_penalty * token_frequency`
   412│ 
   413│ **Presence Penalty (存在惩罚)：**
   414│ - 取值范围：-2.0 到 2.0
   415│ - 正值：鼓励引入新主题（只要Token出现过一次就惩罚）
   416│ - 负值：鼓励维持当前主题
   417│ 
   418│ **参数组合策略表：**
   419│ 
   420│ | 任务类型 | Temp | Top-p | Freq Penalty | Pres Penalty |
   421│ |---------|------|-------|--------------|---------------|
   422│ | 事实问答 | 0.1 | 0.9 | 0.0 | 0.0 |
   423│ | 摘要生成 | 0.3 | 0.95 | 0.5 | 0.5 |
   424│ | 故事续写 | 0.8 | 0.92 | 0.7 | 0.7 |
   425│ | 代码补全 | 0.2 | 0.85 | 0.0 | 0.0 |
   426│ | 翻译 | 0.4 | 0.95 | 0.0 | 0.0 |
   427│ | 头脑风暴 | 1.0 | 0.98 | 1.0 | 1.0 |
   428│ 
   429│ > **专家提示**：Temperature和Top-p不要同时调高。高温(>0.8)+高Top-p(>0.95)会产生极其混乱的输出。推荐策略：固定Top-p=0.9-0.95，只调节Temperature，或者固定Temperature=0.8-1.0，调节Top-p。
