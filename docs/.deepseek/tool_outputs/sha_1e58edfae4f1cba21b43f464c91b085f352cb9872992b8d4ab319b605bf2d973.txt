### 知识点1: 向量数据库与 Redis 的核心定位与数据模型区别
向量数据库（如 Milvus、Pinecone、Weaviate、Qdrant）是专门为高维向量嵌入的存储和相似性搜索而设计的数据系统，核心数据单元是向量（embedding）；Redis 原本是高性能的内存键值存储，数据模型为 key-value，并扩展了 Hash、List、Set、Sorted Set、Stream 等多种数据结构，擅长高速读写、缓存、消息队列、排行榜等场景。

### 知识点2: 索引与查询方式的根本差异
向量数据库内置多种近似最近邻（ANN）索引算法（如 HNSW、IVF、PQ），深度优化向量搜索，能处理数亿甚至更多高维向量，提供毫秒级相似度检索，查询语言支持向量相似度计算（余弦、欧氏距离等）与标量过滤的混合查询。原生 Redis 没有向量概念，只能通过 key 精确匹配或数值范围查询（如 Sorted Set 的 score），无法进行语义相似性搜索；Redis 通过 RediSearch 模块可支持 FLAT 和 HNSW 索引做 KNN 相似度查询，但向量索引是作为模块附加在键值模型之上的。

### 知识点3: 功能完整性与易用性对比
专业向量数据库提供多种距离度量、索引类型选择及自动调优，高级数据管理（分区、分片、增量插入、数据版本、回滚），混合查询（元数据过滤 + 向量相似度且可在索引层面做剪枝优化），嵌入模型集成，以及与 AI 生态（LangChain、LlamaIndex 等）的对接能力。Redis + RediSearch 的向量功能更像轻量级扩展，混合过滤优化程度、索引构建灵活性、向量维度上限、大规模下的稳定性与专有向量数据库相比仍有差距，但胜在极低延迟和与现有 Redis 基础设施的无缝结合。

### 知识点4: 持久化与存储结构差异
Redis 是纯内存数据库（虽有持久化，但数据主要在内存中服务），向量索引也需加载到内存，处理超大向量数据时成本非常高。许多向量数据库采用内存 + 磁盘混合存储或纯磁盘方案，能更经济地存储百亿级向量，并通过缓存机制保持高性能。

### 知识点5: 适用场景对比
使用 Redis（含向量搜索）的场景：已深度使用 Redis 的团队，需要快速给现有应用增加语义搜索、推荐或 RAG 功能，数据规模可控、运维简单优先，例如为 Session 缓存数据做实时相似商品推荐。使用专门向量数据库的场景：核心业务依赖大规模向量搜索（如十亿级相似图片检索、长文档语义索引）、需要极致搜索性能与调优、复杂的多租户隔离、以及完善的 AI 工具链集成。

### 知识点6: 系统组件依赖（部署复杂度）
Redis 依赖极简，是单一二进制程序，几乎没有外部依赖，只需启动 redis-server 进程，顶多配置哨兵或集群模式，不依赖第三方存储或协调服务。专业向量数据库（如 Milvus）为了云原生和存储计算分离通常拆成微服务架构，强依赖 etcd（存储元数据）、MinIO/S3（存储原始向量和索引文件）、Pulsar/Kafka（消息队列处理数据变更），部署维护重。也有 Qdrant、Weaviate 等可单二进制运行的轻量选择，但内部仍可能嵌入 RocksDB 等存储引擎。
