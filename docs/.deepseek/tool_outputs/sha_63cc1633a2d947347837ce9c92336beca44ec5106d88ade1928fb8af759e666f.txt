    47│ ### 二、 重构方案总览：模块化分层架构
    48│ 
    49│ 建议将项目重构为 **四层架构 + 公共基础设施** 结构。
    50│ 
    51│ ```text
    52│ ExcelAnalyzer/
    53│ ├── app/                          # 应用层 (启动与组装)
    54│ │   ├── main.py                   # 入口文件（只负责 QApplication 和主窗口启动）
    55│ │   └── hooks/                    # PyInstaller 运行时钩子（替代散落的 .py 文件）
    56│ │
    57│ ├── ui/                           # 视图层 (纯 UI 生成与信号定义)
    58│ │   ├── windows/                  # 主窗口、对话框 UI 类（由 Designer 生成或手写）
    59│ │   ├── widgets/                  # 自定义控件（如依赖树控件）
    60│ │   └── resources/                # 图标、qss 样式
    61│ │
    62│ ├── controllers/                  # 控制器层 (响应 UI 事件，调度业务逻辑)
    63│ │   ├── main_controller.py        # 协调各子控制器
    64│ │   ├── template_controller.py
    65│ │   ├── analysis_controller.py
    66│ │   └── batch_controller.py
    67│ │
    68│ ├── services/                     # 业务逻辑层 (核心功能实现，与 UI 解耦)
    69│ │   ├── parser/                   # 解析服务（封装 VBA/COM/OpenPyXL 切换逻辑）
    70│ │   │   ├── base.py               # 解析器抽象基类
    71│ │   │   ├── vba_adapter.py        # VBA 注入与解析实现
    72│ │   │   └── fallback_adapter.py   # OpenPyXL 备选方案
    73│ │   ├── analyzer/                 # 依赖分析、格式提取
    74│ │   ├── template_manager/         # 模板数据库操作
    75│ │   └── batch_processor/          # 批量处理任务队列
    76│ │
    77│ ├── infrastructure/               # 基础设施层 (跨业务的技术支撑)
    78│ │   ├── com_bridge/               # COM/WPS 连接池、生命周期管理（修复 COM 线程问题）
    79│ │   ├── logger/                   # 结构化日志（JSON 格式，便于排查 VBA 错误）
    80│ │   ├── config/                   # 配置管理（读取 config.json）
    81│ │   └── events/                   # 全局事件总线（解耦控制器间通信）
    82│ │
    83│ ├── data/                         # 数据层 (ORM/DAO 定义)
    84│ │   └── models/                   # SQLite 表结构定义
    85│ │
    86│ ├── vba_modules/                  # VBA 脚本资源（保持不变，但由 Services 层读取）
    87│ ├── tests/                        # 测试目录（替代根目录散落的 test_*.py）
    88│ │   ├── unit/
    89│ │   └── integration/
    90│ │
    91│ ├── build/                        # 构建配置
    92│ │   ├── ExcelAnalyzer.spec        # 精简后的 spec
    93│ │   └── hooks/                    # 自定义 PyInstaller hooks（解决 pandas 依赖）
    94│ │
    95│ └── pyproject.toml                # 项目元数据与依赖声明 (替代 requirements.txt 散落)
    96│ ```

   100│ ### 三、 详细重构步骤与策略
   101│ 
   102│ #### 1. 代码规范化与路径清洗 (Day 1-2)
   103│ - **动作**：全局替换 `from ..xxx` 为 `from services.xxx` 或 `from infrastructure.xxx`。
   104│ - **策略**：修改 `sys.path` 使得项目根目录 `ExcelAnalyzer` 为唯一 Python 根路径。
   105│ - **解决痛点**：彻底消灭 `ModuleNotFoundError`。
   106│ 
   107│ #### 2. 基础设施层重建：高可用 COM/VBA 桥接器 (Day 2-4)
   108│ - **问题聚焦**：文档中频繁出现 WPS ProgID 找不到、线程未初始化、JIT Debugger 弹窗。
   109│ - **重构方案**：
   110│     1.  实现 `infrastructure/com_bridge/office_application.py`，封装 **单例连接池**。
   111│     2.  **自动修复机制**：若 `Ket.Application` 失败，自动尝试 `Excel.Application`，并记录到结构化日志。
   112│     3.  **强制线程安全**：在任何调用 COM 的线程中显式调用 `pythoncom.CoInitialize()`，利用 `with` 上下文管理器确保释放。
   113│     4.  **VBA 模块加载优化**：将 VBA 代码的读取与注入剥离。VBA 字符串应在 `services` 层构建，`infrastructure` 层只负责 `Execute`。
   114│ 
   115│ #### 3. 打包配置工程化 (Day 4-5)
   116│ - **问题聚焦**：`spec` 文件中 `hiddenimports` 列表过于冗长，缺乏维护性。
   117│ - **重构方案**：
   118│     1.  **编写 PyInstaller Hook**：创建 `build/hooks/hook-pandas.py`，利用 `collect_submodules` 自动收集 `pandas._libs`，而不是手动列举。
   119│     2.  **Tree Shaking**：利用 `--exclude-module` 明确排除 `matplotlib`、`scipy` 和未使用的 `PyQt5` 组件（如 `QtWebEngine`）。
   120│     3.  **数据文件映射**：使用 `--add-data` 将 `vba_modules` 和 `config.json` 精确映射到包内路径。
   121│ 
   122│ #### 4. 业务逻辑服务化：剥离 VBA 解析与模板管理 (Day 5-7)
   123│ - **问题聚焦**：`BatchController` 仍直接依赖具体的 VBA 执行代码，`MainWindow` 存在 `findChild` 查找控件的脆弱逻辑。
   124│ - **重构方案**：
   125│     1.  **抽象解析服务**：定义 `IParser` 接口。`MainWindow` 只调用 `self.parser_service.parse(filepath)`，不关心底层是 VBA 还是 OpenPyXL。
   126│     2.  **事件总线解耦**：使用 `infrastructure/events` 发布 `ParseProgressEvent`，`UIStateController` 订阅该事件更新进度条，消除 UI 对具体业务线程对象的直接依赖（修复 `int object has no attribute connect` 错误）。
   127│ 
   128│ #### 5. 测试体系建设 (Day 7-8)
   129│ - **动作**：
   130│     1.  迁移根目录所有 `test_*.py` 到 `tests/unit`。
   131│     2.  编写针对 `OfficeConnector` 的 Mock 测试，避免测试时必须依赖真实 WPS 环境。
   132│     3.  增加针对 `VBAInjector.extract` 返回数据结构的单元测试。

   146│ ### 五、 执行建议
   147│ 
   148│ 建议按照以下顺序逐步重构，**严禁一次性推翻重写**：
   149│ 
   150│ 1.  **分支策略**：创建 `refactor/architecture-v2` 分支。
   151│ 2.  **基础设施先行**：先重构 `infrastructure` 层，确保 COM/VBA 连接比原来更稳定。
   152│ 3.  **渐进替换**：在 `main_window.py` 中逐步将 `self.batch_controller` 的调用替换为新的 `Service` 调用。
   153│ 4.  **旧代码标记**：将 `modules/` 目录标记为 `legacy`，新功能不在其中开发，逐步将其中逻辑迁移至 `services` 并删除。

   155│ 此重构方案将解决文档中记录的所有顽固问题，使项目具备长期迭代的生命力。

   179│ ### 一、 重构背景与必要性分析
   180│ 
   181│ 根据《测试报告》、《修复记录》及《打包命令》等文档分析，当前项目存在以下核心矛盾与风险：
   182│ 
   183│ 1.  **架构混杂与回退逻辑**：
   184│     - `main_window.py` 中存在传统 `ThreadedExcelParser` 回退代码。若VBA/WPS连接失败，程序会静默降级，**绕开了WPS环境要求**，导致用户无法感知功能缺失的真正原因。
   185│ 2.  **WPS适配不彻底**：
   186│     - 虽然 `wps_connector.py` 已增强ProgID检测（`Ket.Application`），但业务逻辑层未强制锁定WPS。COM连接可能意外连接到后台运行的Excel实例，违反“WPS专用”的设计目标。
   187│ 3.  **测试绕过风险**：
   188│     - 历史测试中“VBA数据提取: FAIL”主要因路径问题，但后续验证未严格在**纯净WPS环境**（无Excel安装）下进行，存在测试环境与目标环境不一致的隐患。
   189│ 4.  **打包体积与依赖冗余**：
   190│     - Spec文件中包含大量pandas、matplotlib、scipy的排除/包含逻辑，且部分依赖（如xlrd）并非VBA提取所必需，增加了打包体积和运行时风险。
   191│ 
   192│ ### 二、 重构目标与核心原则
   193│ 
   194│ 1.  **强制WPS锁定**：移除所有Excel COM回退代码，程序启动必须检测到 `Ket.Application` 或 `ET.Application`，否则拒绝运行并给出明确安装指引。
   195│ 2.  **纯VBA解析策略**：彻底废弃 `openpyxl`/`pandas` 的格式解析主路径，所有数据提取（单元格、格式、页眉页脚、依赖）强制通过 `ThreadedVBAParser` + `VBAInjector` 完成。
   196│ 3.  **架构纯净度**：巩固MVC分层，确保 `BatchController` 等业务逻辑不包含任何降级逻辑，且完全基于WPS COM接口实现。
   197│ 4.  **打包轻量化**：精简 `hiddenimports`，仅保留VBA注入和PyQt5核心依赖，剔除不必要的科学计算库。
   198│ 5.  **测试环境强制隔离**：构建无Microsoft Excel的Windows沙箱或虚拟机进行验收测试。
