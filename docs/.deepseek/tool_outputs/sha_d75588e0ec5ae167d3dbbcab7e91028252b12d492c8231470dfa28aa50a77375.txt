  2023│ - **[User Namespace 与 UID 映射机制]** — 容器内独立用户身份的实现原理，确保容器内 root ≠ 宿主机 root
  2024│   - 来源文件：容器与虚拟机区别解析.md
  2025│   - 核心解释：User Namespace 允许将容器内的 UID/GID 映射为宿主机上不同的 UID/GID。Docker 的 `userns-remap` 参数从 `/etc/subuid` 读取映射配置，将容器内 UID 0~65535 线性映射到宿主机 UID 100000~165535。效果：容器内 `whoami` 显示 root（UID=0），但宿主机视角该进程真实 UID 是 100000（普通用户），即使容器被攻破也无法破坏宿主机。注意：Docker 默认未启用 User Namespace，需显式配置。

  2027│ - **[容器内普通用户的宿主机映射规则]** — 在启用 User Namespace 的前提下，容器内 UID x 对应宿主机 UID = base + x
  2028│   - 来源文件：容器与虚拟机区别解析.md
  2029│   - 核心解释：在启用 User Namespace 且配置线性映射（如 base=100000, range=65536）的前提下，容器内 UID x 对应宿主机 UID = base + x。例如容器内普通用户（UID=1001）在宿主机上对应 UID = 101001。这种设计实现了多层隔离：不同容器的 user namespace 独立，容器 A 和 B 的 UID=0 映射到宿主机上是不同的普通用户，互不干扰。
