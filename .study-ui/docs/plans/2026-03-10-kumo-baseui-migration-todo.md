# Study UI Kumo/Base UI Migration Todo Plan

> **For Claude / Codex / other LLM:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` to execute this plan one checklist item at a time. Mark a box only after command or UI evidence. If that skill is unavailable, emulate the same controller + reviewer workflow.

**Goal:** Upgrade the current Claude Code-authored `study-ui` interface to a Kumo-first component system while preserving the custom Tauri desktop shell and using Base UI only where Kumo does not cleanly cover the need.

**Architecture:** Keep the desktop shell custom because the current app owns macOS overlay titlebar behavior, Windows frameless controls, resize handles, and a Tauri-specific sidebar toggle placement. Migrate reusable content controls to Kumo first, then use `@base-ui/react` only as an escape hatch for component gaps or interaction patterns Kumo does not cover well.

**Tech Stack:** React 19, Vite 7, Tailwind CSS 4, Tauri 2, `kumo-ui`, `@base-ui/react`, `@phosphor-icons/react`

---

## Four-Track Research Summary

### Track 1: Product boundary
- Kumo’s official installation docs state that Kumo is built on Base UI and re-exports 37 Base UI primitives.
- Decision: do not run Kumo and Base UI as two competing top-level design systems. Use Kumo as the default component layer and Base UI as a fallback layer.

### Track 2: Coverage vs current repository
- Kumo officially offers styled components and a few higher-level blocks, but not a full Tauri/macOS desktop shell.
- This repository currently mixes shell concerns and content concerns inside `src/App.tsx`.
- Decision: separate shell from content before replacing primitives.

### Track 3: Tooling and styling
- Kumo’s official setup requires installing `kumo-ui`, `@base-ui/react`, and `@phosphor-icons/react`.
- Kumo also requires adding an `@source` entry and importing Kumo styles before Tailwind.
- Decision: bootstrap styles and CLI integration before any component migration.

### Track 4: LLM execution safety
- Kumo exposes a CLI and AI-friendly registry, while Base UI positions itself as an unstyled low-level primitive layer.
- Decision: use Kumo CLI for canonical components and keep direct Base UI usage isolated behind local wrappers.

## Recommended Target State
- **Custom shell layer:** `src/components/shell/AppChrome.tsx`, `src/components/shell/Titlebar.tsx`, `src/components/shell/Sidebar.tsx`, `src/components/shell/WindowControls.tsx`, `src/components/shell/FramelessResizeHandles.tsx`
- **Kumo content layer:** buttons, switches, tabs, dialogs, inputs, surfaces/cards, settings panels, reusable content sections
- **Base UI escape hatch:** only for interaction patterns Kumo does not cleanly cover
